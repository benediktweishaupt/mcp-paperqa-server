# PaperQA2 Full Leverage — PRD for mcp-paperqa-server

Last updated: 2026-03-04

## Overview

MCP server built on PaperQA2 for PhD students in humanities and cultural studies working with scanned books and cultural theory texts (e.g. Hito Steyerl, Luhmann, critical theory).

This is NOT a science-journal workflow. We work with scanned books, complex layouts, footnote-heavy texts, and specialized cultural/theoretical vocabulary.

## Current State (after 2026-03-04 update)

### What's Implemented
- PaperQA2 2026.3.3 on Python 3.12
- `agent_query()` + `get_directory_index()` for search and indexing
- 6 MCP tools: `search_literature` (fast/thorough modes), `get_contexts`, `get_library_status`, `add_document`, `remove_document`, `check_indexing_status`
- Chunk size 6000 / overlap 400 (tuned for book-length texts)
- MMR diversity enabled (`texts_index_mmr_lambda=0.5`)
- Session transparency: cumulative cost, tool history, token counts in responses
- Background indexing via `asyncio.create_task()` for add_document
- `evidence_skip_summary=True` (direct quotes, no paraphrasing)
- `use_doc_details=False` (no journal metadata for books)
- `build_index.py` using shared config

### What's Blocked by Rate Limits
- **Voyage embeddings**: code is ready (`voyage/voyage-3` in config comments), but Voyage free tier has 3 RPM / 10K TPM — unusable for indexing without adding a payment method at https://dashboard.voyageai.com
- **Multimodal parsing**: code is ready (`multimodal=True` in ParsingSettings), but image enrichment makes GPT-4o calls per image, which hits the 30K TPM limit on low-tier OpenAI accounts
- **Hybrid search**: requires Voyage first — the `"hybrid-"` prefix works with any embedding model but only makes sense with Voyage's 32K context

### What We Correctly Skip
- Semantic Scholar, CrossRef, OpenAlex, Unpaywall (journal-paper metadata — irrelevant for scanned books)
- Journal quality scoring (no journals)
- Retraction checking (books don't get retracted)
- Clinical trials search (not our domain)
- `use_doc_details=True` (external API calls for metadata we don't need)
- External paper discovery (library is curated, not discovered)
- Agent type changes (LDP ReAct, MemoryAgent etc.)

## Implemented Features

### Feature 1: Chunk Size Tuning — DONE
- `reader_config={"chunk_chars": 6000, "overlap": 400}` (was 4000/200)
- PaperQA2 API changed: `chunk_size`/`overlap` → `reader_config` dict
- `pdfs_use_block_parsing` does not exist as a ParsingSettings attribute in 2026.3.3. Block parsing may be the default or handled differently now. Not investigated further.

### Feature 2: Embedding Model — PARTIALLY DONE
- Code ready for `voyage/voyage-3` (one line change in config.py)
- Kept `text-embedding-3-small` as default because Voyage free tier rate limits make indexing fail
- **Action needed**: add payment method on Voyage dashboard, then change embedding in config.py and rebuild index
- Removed broken `configure_embedding` tool — embedding is a config-time decision

### Feature 3: Hybrid Search — BLOCKED
- `HybridEmbeddingModel` exists in PaperQA2 2026.3.3 (verified)
- The `"hybrid-"` prefix on embedding model name enables it (e.g. `"hybrid-voyage/voyage-3"`)
- Blocked on Voyage rate limits (same as Feature 2)
- **Action needed**: same as Feature 2 — once Voyage works, change embedding to `"hybrid-voyage/voyage-3"`

### Feature 4: MMR Diversity — DONE
- `texts_index_mmr_lambda=0.5` configured at Settings level
- 0.0 = max diversity, 1.0 = pure relevance
- 0.5 balances relevance with diversity across chapters/books
- Field name is `texts_index_mmr_lambda` (not on AnswerSettings)

### Feature 5: Document Management — DONE
- `add_document`: copies file to papers/, starts background indexing via `asyncio.create_task()`, returns job ID
- `remove_document`: deletes file, index refreshes on next search
- `check_indexing_status`: polls in-memory job dict
- Supports PDF, TXT, MD, HTML, DOCX
- Background indexing solves the original MCP timeout problem

### Feature 6: Query Modes — DONE
- `search_literature` accepts `mode="fast"` (evidence_k=3) or `mode="thorough"` (evidence_k=15)
- `max_sources` parameter overrides the mode default

### Feature 7: Session Transparency — DONE
- Cumulative cost tracking across queries
- `tool_history` extracted from `result.session` (guarded with `hasattr`)
- `token_counts` extracted from `result.session` (guarded with `hasattr`)
- Session total shown in response footer

### Feature 8: get_contexts Rewrite — PARTIALLY DONE
- Still uses `agent_query()` internally (not zero-cost)
- `Docs.retrieve_texts()` exists but requires loading all docs via `aadd()` first — too slow per-call
- `SearchIndex.query()` exists for direct Tantivy search but returns raw index results, not Text objects with citations
- Current approach: agent_query with `evidence_skip_summary=True` and configurable `evidence_k`
- **Future improvement**: find a way to use the built index directly without loading all docs

## Remaining Work

### Tier 1: Unblock When Rate Limits Solved

These are ready in code, just need higher API tier:

1. **Voyage embeddings** — change `embedding="voyage/voyage-3"` in config.py, rebuild index
   - Prerequisite: add payment method on Voyage dashboard
   - Expected: +7.5% retrieval quality, 32K context (vs 8K), 2x cheaper

2. **Hybrid search** — change `embedding="hybrid-voyage/voyage-3"` in config.py, rebuild index
   - Prerequisite: Voyage working
   - Expected: better recall on exact terminology ("Verfremdungseffekt", "proxy politics")

3. **Multimodal parsing** — set `multimodal=True` in config.py, rebuild index
   - Prerequisite: higher OpenAI tier (>30K TPM) or switch to cheaper LLM for enrichment
   - Expected: image/figure descriptions indexed alongside text

### Tier 2: Further Optimization

4. **True zero-cost get_contexts** — replace agent_query with direct vector search
   - The `SearchIndex.query()` method works but returns raw dicts, not structured Text objects
   - Needs a lightweight adapter to format results with citations
   - Would drop cost from ~$0.05 to ~$0.001 and latency from 30s to <3s

5. **Smarter LLM selection for indexing** — use a cheaper model (gpt-4o-mini) for citation extraction during indexing
   - Currently uses gpt-4o for everything including indexing, which is expensive and hits rate limits
   - PaperQA2 supports `summary_llm` for cheaper operations — investigate

6. **Rate limit resilience** — add retry logic with backoff for indexing
   - PaperQA2's built-in retries (3x via LiteLLM) are not enough for 30K TPM
   - Consider using `max_concurrent_requests=1` + explicit delays between chunks

## API Findings (from 2026-03-04 implementation)

Things we discovered about PaperQA2 2026.3.3 that weren't in the docs:

- `ParsingSettings.chunk_size` and `.overlap` are gone → use `reader_config={"chunk_chars": N, "overlap": N}`
- `ParsingSettings.multimodal` defaults to `1` (enabled) — disabling requires explicit `multimodal=False`
- `Settings.texts_index_mmr_lambda` is the MMR control (not on AnswerSettings or VectorStore)
- `Settings.batch_size` controls document processing parallelism
- `PQASession` fields: `id`, `question`, `answer`, `raw_answer`, `answer_reasoning`, `has_successful_answer`, `context`, `contexts`, `references`, `formatted_answer`, `graded_answer`, `cost`, `token_counts`, `config_md5`, `tool_history`
- `SearchIndex.query(query, top_n, offset, min_score)` exists for direct Tantivy search
- `HybridEmbeddingModel` exists and can be triggered via `"hybrid-"` prefix on model name
- `pdfs_use_block_parsing` does NOT exist as a ParsingSettings attribute
- Indexing makes GPT-4o calls for citation extraction even with `use_doc_details=False`

## Risks

### Rate Limit Ceiling
- **Problem**: OpenAI 30K TPM and Voyage free tier 3 RPM/10K TPM block three features
- **Mitigation**: Add payment methods on both dashboards. Voyage has 200M free tokens even with payment method added — it's just for rate limit unlock.

### Index Rebuild Cost
- **Problem**: Switching to Voyage + hybrid + multimodal requires full index rebuild
- **Mitigation**: Do all three at once (one rebuild). Cost is a few cents for embedding + a few dollars for LLM citation extraction on a small library.

### get_contexts Architecture
- **Problem**: Still uses full agent pipeline, no true zero-cost search
- **Mitigation**: Works fine functionally. Optimization can wait. The cost difference matters at scale, not with 2 books.
